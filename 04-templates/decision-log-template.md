# Decision Log Template

## Purpose

Use this template to record decisions made during current-state SOP mapping.

The decision log helps explain why files, names, scope boundaries, assumptions and mapping choices were made.

It should prevent the repo from becoming a set of undocumented choices.

## Instructions

Record a decision when it affects:

- scope
- file structure
- naming
- SOP inclusion or exclusion
- role interpretation
- evidence handling
- future-state boundary
- business case framing
- cross-function synthesis

Keep decisions short, dated and linked to evidence where possible.

## Decision categories

| Category | Description |
|---|---|
| Scope | What is included or excluded from current-state mapping. |
| Naming | How functions, files, SOPs or concepts are named. |
| Structure | How the repository is organised. |
| Evidence | What evidence is accepted, excluded or marked as uncertain. |
| Role interpretation | How position descriptions are interpreted against actual practice. |
| SOP inclusion | Whether an activity should become a SOP. |
| SOP exclusion | Whether an activity should be excluded or deferred. |
| Cross-function synthesis | How patterns are compared across functions. |
| Future-state boundary | What should remain a signal rather than a future-state design. |
| Business case framing | How evidence should support later investment or operating model work. |

## Decision quality prompts

- Is this decision based on actual practice?
- Is this decision based on position description wording only?
- Has the person doing the work validated it?
- Does this decision change SOP scope?
- Does this decision affect future-state interpretation?
- Does this decision create a risk of over-designing?
- Does this decision preserve uncertainty clearly?
- What would cause this decision to be reviewed?
- What evidence would reverse the decision?

## Example decisions to record

| Example decision | Why it may need recording |
|---|---|
| Map actual practice rather than aspirational role scope. | Prevents unperformed work from becoming current-state evidence. |
| Keep the two functions separate during discovery. | Avoids premature merging of data and knowledge work. |
| Exclude broad CX reporting unless validated. | Protects current-state accuracy. |
| Treat omnichannel implications as signals, not requirements. | Prevents premature future-state design. |
| Use draft SOPs until officer validation occurs. | Preserves evidence quality. |
| Use GitHub as the working repository. | Explains collaboration and version-control choice. |
