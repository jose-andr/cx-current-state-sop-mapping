# Evidence Register Template

## Purpose

Use this template to capture evidence collected during current-state SOP mapping.

Evidence should support what is documented in SOPs, pain points, risks, assumptions, decisions and future operating model signals.

The goal is to distinguish confirmed current practice from interpretation, memory, position description wording or future-state aspiration.

## Instructions

Record evidence safely.

Do not include customer personal information, sensitive screenshots or confidential operational details unless they are approved for storage in this repository.

Where evidence cannot be stored directly, record a safe reference to where it exists.

## Evidence types

| Evidence type | Description | Storage guidance |
|---|---|---|
| Interview note | Notes from a walkthrough or discussion. | Store summary only; mark unvalidated until checked. |
| Process observation | Notes from watching the work happen. | Store summary and avoid personal information. |
| Screenshot | Image of system, tracker or tool. | Redact customer and sensitive information. |
| Email example | Example of request, approval or handoff. | Store metadata or summary unless approved and redacted. |
| Tracker | Spreadsheet, queue or manual work list. | Store sample structure or redacted extract only. |
| System record | CRM, Service Directory or tool record. | Store reference only unless approved and redacted. |
| Document | Position description, guide, policy or instruction. | Store link or approved copy if allowed. |
| Report or export | Operational report or data extract. | Store summary or redacted aggregate only. |
| Example case | Representative example of work. | De-identify and summarise. |

## Evidence quality guide

| Quality | Description |
|---|---|
| Strong | Directly shows actual current practice and has been validated with the person doing the work. |
| Medium | Supports the observation but needs further validation or context. |
| Weak | Based on memory, inference, incomplete notes or position description wording only. |
| Not suitable | Contains sensitive information that cannot be safely stored or used. |

## Evidence use prompts

- What claim does this evidence support?
- Does the evidence show actual current practice?
- Is it only evidence of intended role scope?
- Does it show a common case or an exception?
- Has the person doing the work confirmed it?
- Is it recent enough to rely on?
- Does it contain customer information?
- Can it be safely stored in the repository?
- Should it be summarised instead of copied?
- Does it support a SOP, pain point, risk, assumption or future signal?

## Safe evidence handling

Use these rules by default:

- Do not store customer names, addresses, contact details, case numbers or account details.
- Do not store screenshots with visible customer information.
- Do not store private staff comments unless approved.
- Prefer summaries over raw artefacts where sensitivity is unclear.
- Use redacted examples for process evidence.
- Record where evidence exists rather than copying it when storage is not appropriate.
- Mark evidence as unvalidated until reviewed.
