# Role Intent vs Actual Practice

## Purpose

This file explains how the repository distinguishes between formal role intent and actual current practice.

Position descriptions are useful reference documents, but they do not always describe what is happening operationally today.

Some responsibilities may be:

- aspirational
- historic
- partially implemented
- informally performed
- transferred elsewhere
- not yet operationalised

The purpose of current-state SOP mapping is to document the work as it is actually performed.

## Core principle

Map actual practice first.

Use formal position descriptions as reference material, not as the sole source of truth.

## Why this matters

If the repository documents aspirational duties as if they are current practice, it will create a misleading baseline.

A misleading baseline can lead to:

- inaccurate SOPs
- incorrect assumptions about capability
- unclear ownership
- unrealistic improvement plans
- weak business case evidence
- poor future operating model decisions

An honest current-state map is more useful than a polished but inaccurate one.

## Classification model

Use the following categories when reviewing role responsibilities.

| Category | Meaning | Treatment |
|---|---|---|
| Actual practice | The work is performed today | Map as a current-state SOP |
| Partial practice | The work happens sometimes, inconsistently or informally | Map as draft SOP, variation or gap |
| Aspirational role intent | The work appears in the PD but is not performed in practice | Capture as role gap or future capability signal |
| Historic responsibility | The work used to happen but no longer does | Capture only if relevant to current-state understanding |
| Transferred responsibility | The work is now performed by another person, team or system | Capture ownership note and handoff |
| Future opportunity | The work may be needed for the future model | Capture in future operating model signals |

## How to use this file

When reviewing a PD accountability, ask:

1. Is this work performed today?
2. Who performs it?
3. How often does it happen?
4. What triggers it?
5. What system or tool supports it?
6. Is there a documented process?
7. Is it done consistently?
8. Is the responsibility owned, shared or unclear?
9. Is this operational work, support work, reporting work, governance work or future capability work?
10. Should this become a current SOP, a role-gap note or a future capability signal?

## Customer Data and Systems Support note

The broader position description for this function includes responsibilities for customer experience measurement, reporting and analysis.

Those responsibilities should not be treated as current-state SOPs unless confirmed as work that is actually performed today.

For this current-state mapping, the function should focus on the operational work that is actually happening, such as:

- customer record quality
- duplicate customer record identification
- customer record cleansing
- manual de-duplication
- customer data correction
- CRM data maintenance support
- escalation of data quality or system issues

Broader reporting and analysis responsibilities can be captured as role intent, role gap or future capability signals if they are not active practice.

## Customer Focus Information Support note

The Customer Focus Information Support function should also be mapped based on actual practice.

The role may include a wide range of information support, content, communication, CRM support and service knowledge responsibilities.

Each responsibility should be tested against actual work before it becomes a current-state SOP.

## Documentation rule

Do not create a current-state SOP unless the activity is confirmed as real, recurring or operationally meaningful.

If the activity is unclear, create an open question.

If the activity is aspirational, capture it in a role-gap file.

If the activity is important for the future but not current practice, capture it in future operating model signals.

## Suggested evidence sources

Use multiple sources to validate actual practice:

- officer walkthroughs
- process observation
- existing work instructions
- system task lists
- sample anonymised requests
- existing templates
- recurring calendar activities
- stakeholder requests
- escalation examples
- team rituals or stand-ups
- existing reports or logs

Do not use live customer data or personal information as evidence in this repository.

## Output

Each function should include an `actual-practice-summary.md` file that summarises:

- work confirmed as actual practice
- work performed informally
- work performed inconsistently
- work appearing in the PD but not operationalised
- unclear responsibilities requiring follow-up
- future capability signals
