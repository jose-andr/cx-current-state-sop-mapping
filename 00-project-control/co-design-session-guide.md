# Co-design Session Guide

## Purpose

This guide supports structured SOP discovery sessions with the officers who perform the work today.

The goal is to understand actual current practice before designing future improvements.

## Session principles

- Start with the work, not the position description.
- Let the officer describe the process in their own words.
- Capture exceptions, workarounds and judgment calls.
- Do not force a clean process where the real process is messy.
- Separate actual practice from role intent.
- Avoid sensitive customer information.
- Capture future implications without redesigning the function during the session.

## Recommended session length

60 to 90 minutes per function.

Run separate sessions for:

1. Customer Data and Systems Support
2. Customer Focus Information Support

## Participants

Recommended participants:

- officer who performs the function
- facilitator
- note taker
- relevant manager or program lead, optional

Keep the first session small so the officer can describe the work honestly and practically.

## Materials

Prepare:

- current-state SOP template
- SOP inventory template
- exception log template
- pain point log template
- omnichannel implication template
- role intent versus actual practice guide
- relevant position description for reference only

## Session structure

## 1. Opening

Explain the purpose of the session:

The goal is to map how the work happens today, not to audit performance or redesign the role.

Confirm that the session will focus on:

- real tasks
- real triggers
- real systems
- real decision points
- real exceptions
- real pain points
- future implications

## 2. Start with a normal week

Ask:

- What work usually lands on your desk in a normal week?
- What work happens every day?
- What work happens weekly or monthly?
- What work is ad hoc?
- What work is urgent or reactive?
- What work takes more time than people realise?
- What work depends on your personal knowledge?

Capture each activity in the SOP inventory.

## 3. Sort the work

Classify each activity as:

| Category | Meaning |
|---|---|
| Current SOP candidate | Real, recurring and suitable for process mapping |
| Informal practice | Real but undocumented or inconsistent |
| Exception pathway | Happens when the standard path fails |
| One-off activity | Happens rarely or only on request |
| Role gap | Appears to be expected but is not currently performed |
| Future capability signal | Not current practice but may be needed later |

## 4. Select priority SOPs

Choose 2 to 3 activities to map first.

Prioritise work that is:

- high volume
- high risk
- highly manual
- frequently misunderstood
- dependent on one person
- connected to customer experience
- connected to data or knowledge quality
- relevant to omnichannel readiness

## 5. Walk through one activity

For each priority activity, ask:

- What triggers this work?
- Who sends or initiates it?
- What information do you need before you can start?
- What do you check first?
- What system or tool do you open first?
- What steps do you follow?
- What decisions do you make?
- What rules or criteria guide those decisions?
- What happens if the information is missing?
- What happens if something looks wrong?
- Who do you hand off to?
- How do you record that the work is complete?
- How do you know the outcome is correct?

## 6. Map decision points

For each decision, capture:

| Decision | How the decision is made today | Source of rule | Confidence |
|---|---|---|---|
|  |  | Documented / informal / judgment / manager advice / policy / system rule | High / medium / low |

Ask:

- Is this rule written down?
- Is it applied consistently?
- Who confirms the decision if you are unsure?
- What happens if different people interpret it differently?

## 7. Capture exceptions

Ask:

- What commonly goes wrong?
- What information is usually missing?
- What causes rework?
- What causes delay?
- What cases make you pause?
- What cases need escalation?
- What cases do not fit the normal process?
- What do you do when the system does not support what needs to happen?

Capture:

| Exception | What happens today | Impact | Future implication |
|---|---|---|---|
|  |  |  |  |

## 8. Capture systems and tools

Ask:

- What systems do you use?
- What spreadsheets or trackers do you maintain?
- What inboxes, queues or forms do you monitor?
- What documents or instructions do you rely on?
- What information is copied between systems?
- What information is duplicated?
- What system constraints shape the process?

Capture:

| System or tool | Used for | Pain point | Owner |
|---|---|---|---|
|  |  |  |  |

## 9. Capture pain points and risks

Ask:

- What slows the work down?
- What creates avoidable effort?
- What creates customer impact?
- What creates staff impact?
- What creates data or information quality risk?
- What is hard to explain to others?
- What would be difficult if you were away?
- What needs to be fixed before this could scale?

Capture:

| Pain point or risk | Who is affected | Impact | Evidence |
|---|---|---|---|
|  |  |  |  |

## 10. Capture omnichannel implications

Ask:

- What would need to be true for this to work across channels?
- What customer context needs to be preserved?
- What service information needs to be reusable?
- What data needs to be accurate?
- What could be automated safely?
- What should not be automated?
- What would help staff avoid asking customers to repeat themselves?
- What would help customers complete the interaction in their channel of choice?

Capture:

| Current-state observation | Omnichannel implication | Type |
|---|---|---|
|  |  | Data / knowledge / workflow / routing / governance / automation |

## 11. Review the position description

Only after mapping actual work, review the relevant position description.

Ask:

- Which PD responsibilities are active practice?
- Which are partially active?
- Which are not happening?
- Which are performed by someone else?
- Which are future capability signals?
- Which responsibilities no longer reflect the role?

Capture gaps in the relevant function’s `pd-intent-vs-current-reality.md` or `actual-practice-summary.md`.

## 12. Close the session

Confirm:

- priority SOPs mapped
- open questions
- missing evidence
- follow-up actions
- items requiring manager clarification
- items requiring system or policy clarification

## Outputs from each session

Each session should produce or update:

- `sop-inventory.md`
- `actual-practice-summary.md`
- one or more draft SOP files
- `decision-points.md`
- `exception-pathways.md`
- `systems-and-tools.md`
- `pain-points-and-risks.md`
- `omnichannel-implications.md`

## Follow-up

After the session:

- clean up notes
- draft SOPs in plain language
- mark assumptions clearly
- create GitHub issues for unresolved questions
- validate the draft with the officer
- avoid publishing as final until reviewed
