# Working Principles

## 1. Map actual practice first

This repository maps how work is currently performed.

Do not assume that a responsibility listed in a position description is active current practice.

If a responsibility is not performed today, capture it as a role gap, future capability signal or historical note.

## 2. Keep the two functions separate during discovery

Customer Data and Systems Support and Customer Focus Information Support should be mapped separately during the current-state phase.

Do not merge processes, ownership or accountabilities before both functions have been properly understood.

Shared patterns should be captured in `03-cross-function-learning/`.

## 3. Co-design with the officers who perform the work

The officers who run the functions day to day are the source of truth for current practice.

Use position descriptions, system notes and existing documents as reference material, but validate the process with the people performing the work.

## 4. Capture the informal work

Current-state mapping should include work that is undocumented, informal, manual or dependent on individual knowledge.

This includes:

- workarounds
- judgment calls
- escalation habits
- personal checklists
- manual quality checks
- repeated clarifications
- known system constraints
- undocumented decision rules

## 5. Separate facts from assumptions

When something is known, document it as current practice.

When something is uncertain, document it in the assumptions log.

When something needs confirmation, create an open question or GitHub issue.

## 6. Do not over-standardise too early

Some current processes may be unstable, unclear or inconsistent.

Do not force a clean SOP before the actual variation is understood.

Use draft SOPs, exception logs and process notes to capture complexity before simplifying it.

## 7. Design for future reuse without redesigning yet

While the repository is focused on current state, each SOP should capture future implications.

Look for signals related to:

- data quality
- service knowledge quality
- duplicated information
- customer context
- staff guidance
- first contact resolution
- channel continuity
- automation readiness
- enterprise governance

## 8. Keep the repository enterprise-safe

Do not store:

- customer records
- personal information
- credentials
- restricted operational material
- confidential staff information
- screenshots containing sensitive data
- raw extracts from production systems

Use fictional, anonymised or generalised examples only.

## 9. Make ownership visible

Every SOP should identify:

- current owner
- maintainer
- reviewer
- status
- version
- date mapped
- date reviewed

If ownership is unclear, document it as a governance gap.

## 10. Use plain language

SOPs should be readable by non-specialists.

Avoid unnecessary jargon.

Use clear headings, short steps, decision tables and exception tables.

## 11. Keep GitHub changes small and traceable

Use small commits with clear messages.

Examples:

- `Add working principles`
- `Draft CDSS duplicate record SOP`
- `Add CFIS Service Directory update process`
- `Capture role gap for reporting responsibilities`
- `Add data quality exception log`

## 12. Treat SOPs as living artefacts

Current-state SOPs are not final policy.

They should evolve as the process is validated, corrected and improved.

Each SOP should move through clear statuses:

- Draft
- In review
- Validated current state
- Superseded
- Deprecated

## 13. Compare only after mapping

Cross-function comparison should happen after each function has been mapped separately.

The comparison should look for:

- shared systems
- shared handoffs
- shared data dependencies
- shared knowledge dependencies
- duplicated work
- governance gaps
- automation candidates
- future operating model signals

## 14. Use the work to build evidence

The repository should help build an evidence base for future decisions.

The goal is not just to document tasks.

The goal is to understand what CX needs to govern, improve, reuse or automate to support better service delivery.
