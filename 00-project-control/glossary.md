# Glossary

## Purpose

This glossary defines key terms used across the current-state SOP mapping repository.

The purpose is to keep language consistent across both functions and avoid confusion between data, systems, knowledge, content and process terms.

## Terms

| Term | Meaning |
|---|---|
| Current state | How the work is actually performed today. |
| Future state | How the work may operate after improvement, redesign or operating model change. |
| SOP | Standard operating procedure. A documented process that explains how a recurring activity is performed. |
| Current-state SOP | A draft SOP that documents how the process works today, including known exceptions and workarounds. |
| Approved SOP | A formally reviewed and approved procedure used as an operational standard. This repository does not yet contain approved SOPs. |
| Work type | A recurring category of work performed by a function. |
| Trigger | The event, request or condition that starts a process. |
| Input | Information, evidence, request details or system data required to perform a process. |
| Output | The result produced by a process. |
| Decision point | A moment in a process where the officer must choose between different actions or pathways. |
| Exception | A case that does not follow the normal process. |
| Workaround | An informal or manual method used to complete work when the standard process, system or information is insufficient. |
| Handoff | A transfer of work, information, responsibility or decision-making from one person, team or system to another. |
| Control | A check, approval, rule or safeguard that helps ensure quality, accuracy, compliance or consistency. |
| Pain point | A source of friction, delay, confusion, rework or risk. |
| Risk | A potential issue that could affect service quality, data quality, compliance, continuity, staff effort or customer experience. |
| Assumption | Something believed to be true but not yet confirmed. |
| Open question | Something that needs to be clarified before the SOP or process map can be considered reliable. |
| Role gap | A difference between what the role is expected to do and what is actually performed in practice. |
| Capability signal | Evidence that a future skill, role, process, governance model or system capability may be needed. |
| Omnichannel implication | A current-state observation that has relevance for future connected service delivery across channels. |
| Automation candidate | A process step, decision, rule, notification or data movement that may be suitable for automation in future. |
| Source of truth | The authoritative place where a piece of information, data or rule should be maintained. |
| Customer data | Structured information about a customer, account, contact, interaction or relationship. |
| Customer record | A CRM or system record representing a customer, account, contact or related entity. |
| Data quality | The degree to which data is accurate, complete, current, unique, consistent and usable. |
| Duplicate record | Two or more customer records that appear to represent the same person, organisation, account or entity. |
| De-duplication | The process of identifying, reviewing, merging, correcting or resolving duplicate records. |
| Data correction | An action taken to fix inaccurate, incomplete, outdated or inconsistent customer data. |
| CRM | Customer Relationship Management system. In this context, it may refer to Salesforce, Pathway or related customer service systems. |
| Service knowledge | Information that staff, customers or systems need to understand, explain or complete a service interaction. |
| Service Directory | A customer service information source used to support staff in resolving customer enquiries. |
| Customer service tools | Tools used to support customer service delivery, such as Service Directory, Customer Focus Portal, CoMWeb, CRM or related applications. |
| First contact resolution | Resolving a customer enquiry at the first point of contact without unnecessary transfer, repeat contact or rework. |
| Content quality | The degree to which service information is accurate, current, clear, complete, usable and owned. |
| Knowledge owner | The person, role or team accountable for the accuracy and currency of a piece of service knowledge. |
| Content maintainer | The person or role responsible for updating service information in a tool or channel. |
| Editor | A person who updates or maintains content in a service information system. |
| Hot topic | An emerging issue, campaign, media topic or operational event likely to generate customer contact. |
| Channel | A way customers or staff interact with the organisation, such as phone, email, web, chat, in person, social media or internal systems. |
| Channel continuity | The ability for customer context, service context or case information to carry across channels without unnecessary repetition. |
| Customer context | Information about the customer, their account, history, request, case or situation that helps staff or systems provide better service. |
| Information reuse | Designing information so it can be reused across multiple tools, channels, processes or service interactions. |
| Enterprise migration | The future movement of this concept repository into an enterprise-owned and governed environment. |

## Notes

Add terms as they appear during SOP mapping.

If a term is used differently by different teams, document the variation rather than forcing a single definition too early.
