# Role Gap and Capability Signals

## Purpose

This file captures role gaps and capability signals emerging from current-state SOP mapping across:

- Customer Data and Systems Support
- Customer Focus Information Support

The purpose is to identify where actual work reveals capability needs that may not be clearly owned, resourced, governed or designed in the current operating model.

This file supports future synthesis and business case development.

## Current status

Status: Draft discovery placeholder.

Signals should be validated through SOP walkthroughs and discussion with officers, managers and relevant stakeholders.

## Important boundary

This file maps capability signals from actual practice.

It should not assume that a future role, team or structure already exists.

It should not reinstate aspirational role elements that were not performed in practice.

In particular, broad CX measurement, reporting and analysis should only be included if validated as actual current practice.

## Capability areas

## 1. Customer data stewardship

Possible capability need:

- customer data quality standards
- duplicate detection and resolution rules
- data correction pathways
- customer record ownership
- CRM data quality controls
- data quality monitoring
- escalation thresholds for complex records

Current-state evidence to gather:

- examples of duplicate or incorrect records
- workarounds used to correct records
- rules used by officers
- unresolved ownership questions
- repeat causes of data quality issues

## 2. Service knowledge stewardship

Possible capability need:

- service information ownership
- Service Directory content standards
- editor support model
- approval pathways
- content review cycles
- first contact resolution content design
- urgent update and expiry rules

Current-state evidence to gather:

- examples of outdated or unclear content
- examples of duplicated information
- editor support requests
- urgent updates and hot topics
- content gaps that cause escalations

## 3. Request triage and classification

Possible capability need:

- classification of requests by type
- routing rules for data, knowledge, access, system and technical issues
- clear handoff pathways
- issue status visibility
- escalation thresholds
- feedback loops for repeated requests

Current-state evidence to gather:

- examples of misrouted requests
- CRM support requests that combine multiple issue types
- unclear escalation pathways
- handoffs between functions
- unresolved or delayed requests

## 4. Workflow visibility

Possible capability need:

- visible work queues
- update status
- ownership status
- review dates
- approval states
- completion confirmation
- audit or evidence trail

Current-state evidence to gather:

- trackers or spreadsheets used today
- email or Teams-based coordination
- work that is hard to hand over
- tasks that rely on memory
- recurring follow-up effort

## 5. Continuous improvement and prevention

Possible capability need:

- root cause review of repeated data errors
- root cause review of repeated content gaps
- feedback loops from staff questions
- upstream training or guidance improvement
- system improvement recommendations
- prevention controls

Current-state evidence to gather:

- repeated duplicate creation patterns
- repeated CRM user questions
- repeated Service Directory content gaps
- issues that return after correction
- manual fixes that could be prevented upstream

## 6. Omnichannel readiness

Possible capability need:

- clean customer identity
- trusted service knowledge
- reusable content model
- source-of-truth rules
- data and knowledge governance
- human review thresholds
- approval and lifecycle states
- cross-channel consistency controls

Current-state evidence to gather:

- information duplicated across tools
- service guidance used by multiple channels
- customer data issues affecting staff decisions
- content issues affecting first contact resolution
- urgent updates affecting multiple channels

## Boundary questions

Use these prompts to avoid over-designing future roles too early.

- Is this a real current-state activity or an aspirational activity?
- Who actually performs the work today?
- Who is expected to perform the work?
- What happens if nobody performs it?
- Is the gap about capacity, authority, skill, process, governance or system design?
- Is this a role gap or a workflow gap?
- Is this a governance gap or a tooling gap?
- Is this a one-off problem or repeated pattern?
- Is the issue best solved by a role, a rule, a workflow, a system change or training?
- What evidence would support this in a business case?
